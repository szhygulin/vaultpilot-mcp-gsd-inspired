## 4byte API contract (research § Code Example 5 + research § Q9 + concrete API call in research § Sources line 1242)

Endpoint: `https://www.4byte.directory/api/v1/signatures/?hex_signature=0xXXXXXXXX`

Response shape on 200 OK:
```json
{
  "count": 1,
  "next": null,
  "previous": null,
  "results": [
    {
      "id": 31781,
      "created_at": "2018-05-12T22:04:23.682594Z",
      "text_signature": "transfer(address,uint256)",
      "hex_signature": "0xa9059cbb",
      "bytes_signature": "..."
    }
  ]
}
```

Response shape on no match: same shape, `count: 0, results: []`.

Network behaviors to surface verbatim per PREP-06:
- HTTP 5xx → `{ kind: "error", message: "4byte.directory returned HTTP 503" }` (or whatever the status is)
- Timeout (1.5s `AbortController`) → `{ kind: "error", message: "4byte.directory unreachable (timeout 1.5s)" }`
- Network unreachable / DNS fail → `{ kind: "error", message: "4byte.directory unreachable: <upstream error message>" }`
- JSON parse failure → `{ kind: "error", message: "4byte.directory invalid response shape" }`

NEVER mask the error as `not-found` (would silently degrade the cross-check). NEVER throw out of `lookupSelector` — the caller (Plan 04-03) treats this as best-effort and SHOULD NOT have to wrap in try/catch.

## Multi-selector-collision handling (research § Q9)

Some selectors have intentional spam — `0x00000000` returns dozens of entries. Phase 4 returns the FIRST entry's `text_signature`. v1.3 SEC-35 dispatch-target allowlist narrows further (cross-check against known-good ABI). Documented in source comment + tool description.

## LRU cache

Process-local `Map<Hex, FourbyteResult>`. Max 256 entries; evicts oldest via `Map`'s insertion-order iteration. Caches `error` results too — a 1.5s timeout cached for the rest of the process saves hammering a down API. Dies with the process. `_resetFourbyteCacheForTesting()` exports for test isolation.

## get_tx_verification per-status re-emit (research § Q11)

The handler reads `record.status` and assembles the response blocks:

- **`prepared`:** just the `PREPARE RECEIPT` block (re-substituted from `record.args`). No LEDGER block (no preview ran). No 4byte block. Plus a `VERIFY_BEFORE_SIGNING_TEMPLATE` summary noting that preview hasn't run yet.
- **`previewed`:** all the blocks `preview_send` emitted — `PREPARE RECEIPT` + `LEDGER BLIND-SIGN HASH` (from `record.pinned.presignHash`) + `AGENT TASK` (with the SAME substitution as preview emitted) + 4byte block (RE-LOOKUP via `lookupSelector(record.pinned.selector)` — cached, so same result if recent) + `VERIFY_BEFORE_SIGNING_TEMPLATE`.
- **`sent`:** all the previewed blocks + an "already broadcast" line with `record.txHash` + `record.sentAt` ISO timestamp. Useful for the user to copy/paste the txHash to Etherscan.
- **`cancelled`:** all blocks up through preview (if it ran) + a "cancelled at" line with `record.cancelledAt` ISO timestamp.

15-min TTL from `createdAt` — past it, `HANDLE_EXPIRED` envelope.

## Tool description (agent routing prompt)

The description names this as the **context-eviction recovery** tool. The agent calls it when:
1. The conversation has rolled past its context window and the agent has lost the `prepare_native_send` / `preview_send` response.
2. The user is mid-flow ("can you remind me what I'm signing?") and the agent needs to re-emit the verification artifacts.
3. The agent wants a stale-snapshot of the handle's state for debugging.

Use cases that DO NOT route to `get_tx_verification`:
- The agent has the original response in context — read it from there; calling this tool is a context-saving optimization, not a fresh source of truth.
- The user wants to send a different tx — call `prepare_native_send` for a fresh handle.

The tool's failure mode (HANDLE_EXPIRED) is documented prominently — Pitfall 5 from research § Common Pitfalls means a 15-min stale handle is gone forever; user re-runs prepare.
