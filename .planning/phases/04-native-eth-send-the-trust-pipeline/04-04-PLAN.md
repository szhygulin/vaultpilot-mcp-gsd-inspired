<objective>
Land the `send_transaction` MCP tool — the third and final step of the trust pipeline. The handler enforces three gates in order: (1) schema-level `previewToken` + `userDecision` enum gate at the MCP boundary (PREP-07 — caught by ajv BEFORE the handler runs); (2) state-machine + previewToken-value match in the handler (`PREVIEW_REQUIRED` / `PREVIEW_TOKEN_MISMATCH`); (3) `payloadFingerprint` drift re-check (PREP-08). After all three gates pass, the handler forwards an `eth_sendTransaction` request over the WC session to Ledger Live → device, which signs AND broadcasts internally and returns the broadcasted `txHash`. Also lands the one-line `getActiveSessionTopic` export on `src/wallet/session-manager.ts` (Q2 locked decision) plus the `test/trust-pipeline.integration.test.ts` end-to-end test that anchors the whole pipeline against the Fixture A+B+C hex values from research § Code Examples.

Purpose: This is the last server-side defense before the bytes hit the device. The drift re-check (PREP-08) is the load-bearing guard against in-process state corruption between prepare and send. The schema-level `userDecision` enum is the load-bearing guard against handler-side soft-check bugs. The integration test is the project's load-bearing assertion that the entire trust pipeline holds end-to-end — every byte the mock device "signs" matches the byte the mock MCP prepared.

Output: 1 extended source file (`session-manager.ts` +1 line) + 1 new tool file + 1 added import in `register-all.ts` + 1 new unit test file + 1 new integration test file (~15 cases total).

## EIP-1559 param encoding for `eth_sendTransaction`

Per WC spec + JSON-RPC convention, all numeric fields are `0x`-prefixed hex (NOT JS numbers, NOT bigints — bigint is not JSON-safe). Use `viem.toHex(bigint)`:

```typescript
import { toHex } from "viem";

const txParams = [{
  from: status.address,
  to: record.tx.to,
  value: toHex(record.tx.valueWei),                        // bigint → "0xde0b6b3a7640000"
  gas: toHex(record.pinned.gas),                            // bigint → "0x5208"
  maxFeePerGas: toHex(record.pinned.maxFeePerGas),          // bigint → "0x6fc23ac00"
  maxPriorityFeePerGas: toHex(record.pinned.maxPriorityFeePerGas),  // bigint → "0x59682f00"
  nonce: toHex(record.pinned.nonce),                        // number → "0x7"
  data: record.tx.data,                                     // already 0x-hex
}];
```

`toHex` accepts both `bigint` AND `number` — confirmed at `viem/_types/utils/encoding/toHex.d.ts:91`.

## Locked decision Q1: `userDecision` enum is `["send", "cancel"]` — handler routes cancel cleanly

From research § Open Questions Q1: expand the enum to `["send", "cancel"]`. Handler routes `"cancel"` to a structured `{ userCancelled: true }` non-error response. Handle transitions to terminal `"cancelled"` then is evicted on TTL.

```typescript
const INPUT_SCHEMA = {
  type: "object" as const,
  properties: {
    handle: { type: "string" },
    previewToken: { type: "string", description: "previewToken returned by preview_send; single-use per pin." },
    userDecision: {
      type: "string",
      enum: ["send", "cancel"],
      description: "Must be exactly \"send\" to broadcast, or \"cancel\" for a clean exit (handle transitions to terminal cancelled state).",
    },
  },
  required: ["handle", "previewToken", "userDecision"],
  additionalProperties: false,
};
```

The MCP SDK's ajv validator enforces `enum: ["send", "cancel"]` at the protocol boundary BEFORE the handler runs (research § Pattern 3 + A7 — `ajv` is a direct dep of `@modelcontextprotocol/sdk@1.29.0`). Any other value (e.g. `"yes"`, `"approve"`, `""`, `null`, missing) rejects at the boundary with a JSON-schema validation error; the handler is never invoked. This is the schema-level gate PREP-07 mandates — NOT a soft `if` in the handler.

## DEMO-05 (anticipated): demo-mode `send_transaction` is a SIMULATION, not refusal

Unlike `prepare_native_send` + `preview_send` (which refuse in demo mode), `send_transaction` in demo mode runs the unsigned tx through `eth_call` for revert detection and returns a structured simulation envelope. This is the v1.0 Phase 5 contract (DEMO-05); the simulation envelope shape is locked HERE in Phase 4 so Phase 5 doesn't reshape it.

Simulation envelope shape:
```typescript
{
  isError: false,
  content: [{ type: "text", text: "SIMULATION (demo mode)\n  to: ...\n  value: ...\n  result: ... (no broadcast performed)" }],
  structuredContent: {
    simulated: true,
    simulationResult: <hex returned by eth_call, or null on revert>,
    simulatedAt: <ISO timestamp>,
    handle: <handle>,
    chainId: 1,
    nonce, gas, maxFeePerGas, maxPriorityFeePerGas,   // pinned values, surfaced for parity with real-broadcast response
  },
}
```

Demo-mode `userDecision: "cancel"` STILL routes to the `userCancelled` envelope (same as production); the demo simulation only fires on `userDecision: "send"`.

Phase 5 evolves the implementation (better persona-aware mocks, fee market simulation, etc.) but the envelope shape is locked here.
