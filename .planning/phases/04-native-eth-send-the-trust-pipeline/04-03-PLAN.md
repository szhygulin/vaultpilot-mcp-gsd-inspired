<objective>
Land the `preview_send` MCP tool — the second step of the trust pipeline. The handler reads a prepared handle, resolves and pins `nonce` / `gas` / `maxFeePerGas` / `maxPriorityFeePerGas` AT PREVIEW TIME (not prepare time — pinning later minimizes the window during which on-chain state drifts from the bytes the user approves), recomputes the EIP-1559 pre-sign hash via Plan 04-01's pure fn (PREP-04), mints a fresh UUID `previewToken`, and emits three plain-text blocks in `content[0].text`: `LEDGER BLIND-SIGN HASH`, `[AGENT TASK — RUN THESE CHECKS NOW]`, and the 4byte cross-check summary.

Purpose: This is where the user's verification ritual actually lives. The user reads the LEDGER BLIND-SIGN HASH block, matches it against the device screen, AND reads the agent's CHECKS PERFORMED block (the agent runs the AGENT TASK checks before asking for confirmation). The 4byte block is the third pillar of cross-verification — best-effort selector lookup with verbatim error surfacing per PREP-06. After this tool succeeds, the user can confidently call `send_transaction` with `{ handle, previewToken, userDecision: "send" }`.

Output: 1 new tool file + 1 added import in `register-all.ts` + 1 new test file (~10 cases including idempotent re-preview, 4byte not-applicable path, fixture-anchored presign hash). No new dep.
</objective>

## Locked decision: idempotent re-preview (Q4 from research § Open Questions)

A second call to `preview_send(handle)` on a `previewed` handle is ALLOWED. Each call:
1. Re-resolves fresh nonce/gas/fees from the RPC (the user may have paused for minutes; on-chain state may have changed).
2. Mints a fresh `previewToken` UUID.
3. Re-pins via `transitionToPreviewed` (which overwrites `record.pinned` — Plan 04-01 specifies this).
4. **The prior `previewToken` is no longer valid** — only `record.pinned.previewToken` (the most recently minted) matches at send time. Plan 04-04's send-time check rejects stale tokens with `PREVIEW_TOKEN_MISMATCH`.

Source comment in the handler:
```typescript
// Idempotent re-preview per locked decision Q4 (research § Open Questions).
// Rationale: gas/nonce/fees go stale over time. If the user pauses 10 min after
// reading the LEDGER BLIND-SIGN HASH, the agent can call preview_send again to
// freshen the pin without forcing a re-prepare (which would change the
// payloadFingerprint and break the trust binding).
// Caveat: a fresh previewToken INVALIDATES the prior one. The handle's pinned
// state is the SOT — only the most-recently-minted token matches at send time.
```

## 4byte block format

For native send (`data === "0x"`, selector === null), the block shows `"not-applicable"` verbatim. For a contract call (Phase 6+), the block shows the discovered text signature or the error message. Block layout:

```text
4BYTE CROSS-CHECK
  Selector:  {SELECTOR_OR_NA}
  Decoded:   {TEXT_SIG_OR_STATE}
```

Substitutions for each `FourbyteResult.kind`:
- `not-applicable` → `Selector: (none — native send)`, `Decoded: not-applicable`
- `found` → `Selector: 0x{8-hex}`, `Decoded: {textSignature}`
- `not-found` → `Selector: 0x{8-hex}`, `Decoded: (not found in 4byte.directory)`
- `error` → `Selector: 0x{8-hex}`, `Decoded: error: {message}` ← VERBATIM message per PREP-06

The block is inlined directly in `preview_send.ts` (not a template in `blocks.ts`) because it's a per-tool concern; Phase 6's `prepare_token_send` will emit the same shape with `kind: "found"` for the canonical ERC-20 selectors. Promoting to a template in `blocks.ts` is a Phase 6 concern, not Phase 4 (YAGNI per research § Q14).

## Composite response structure

The handler returns:

```typescript
{
  content: [{
    type: "text",
    text: [
      ledgerBlindSignHashBlock,    // LEDGER_BLIND_SIGN_HASH_TEMPLATE substituted
      "",                           // blank line separator
      agentTaskBlock,               // AGENT_TASK_TEMPLATE substituted
      "",
      fourbyteBlock,                // inlined per the format above
      "",
      VERIFY_BEFORE_SIGNING_TEMPLATE,  // summary — no substitution
    ].join("\n"),
  }],
  structuredContent: {
    previewToken,                    // UUID v4 — agent must pass to send_transaction
    presignHash,                     // Hex — the bytes the device displays
    chainId: 1,
    nonce, gas, maxFeePerGas, maxPriorityFeePerGas,  // pinned values
    selector,                        // Hex | null
    fourbyte: { kind, ... },         // structured 4byte result for machine-readable consumers
  },
}
```

Block order matters — the user reads top-to-bottom: LEDGER hash (match on device first) → AGENT TASK (what the agent did) → 4byte (independent cross-check) → VERIFY summary (the final pre-confirm reminder).
