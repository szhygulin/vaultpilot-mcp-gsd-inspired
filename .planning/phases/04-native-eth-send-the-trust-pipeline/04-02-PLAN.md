<objective>
Land the `prepare_native_send` MCP tool — the first step of the trust pipeline. The handler validates agent-supplied `{ to, valueWei }`, confirms the Ledger is paired, mints a handle via Plan 04-01's `createHandle`, computes the `payloadFingerprint` (PREP-03) at prepare time, and emits the `PREPARE RECEIPT` block verbatim (PREP-02). NO gas / nonce / fees resolved here — those pin at `preview_send` (Plan 04-03 — pinning at prepare would let on-chain state drift between prepare and preview; research § Anti-Patterns line 416).

The handler's defense-in-depth layering — demo-mode → pairing → input validation → fingerprint compute → store handle — locks the order so future contributors can't accidentally let agent-fabricated `to`/`valueWei` reach state.

Output: 1 new tool file + 1 added import in `register-all.ts` + 1 new test file (~10 cases). No new dep. Tool registered via the Phase 2 side-effect-import pattern.

## Locked decision: `valueWei` is wei-units, NOT decimal-aware

`valueWei` is a STRING (BigInt across the agent boundary is not JSON-safe). Handler parses via `BigInt(args.valueWei)`. Throw structured `INVALID_VALUE` (mapped to `errorCode: "INVALID_INPUT"`) if parse fails. The string represents WEI — NOT human-readable ETH (e.g. `"1000000000000000000"` for 1 ETH; NOT `"1"`). Phase 6 adds decimal-aware parsing for ERC-20 `prepare_token_send` via `get_token_metadata`; that's Phase 6's concern, not Phase 4's. The tool description names this explicitly to prevent agent off-by-decimal confusion.

## Tool description (agent routing prompt — load-bearing for routing correctness)

The description ships as a multi-sentence `.join(" ")` array. Required content (CLAUDE.md "tool descriptions are agent routing prompts"):

1. WHAT it does: "Prepare an unsigned native ETH transfer on Ethereum mainnet. Returns a handle the agent then passes to `preview_send` before `send_transaction`."
2. WHEN to use: "Use when the user wants to send native ETH from their paired Ledger on Ethereum mainnet."
3. WHEN NOT to use:
   - "Do NOT use for ERC-20 token transfers — those land in Phase 6 (`prepare_token_send`)."
   - "Do NOT use for contract calls / approve / swap / withdraw — each gets its own dedicated `prepare_*` tool."
   - "Do NOT use for other chains — v1.0 is Ethereum mainnet only; multi-chain is Phase 8."
4. INPUT discipline:
   - "`valueWei` is the amount in WEI (10^18 wei = 1 ETH), passed as a decimal string (e.g. `\"1000000000000000000\"` for 1 ETH). Do NOT pass human-readable ETH amounts — off-by-decimal is the most common user-facing bug class."
   - "`to` is the recipient address as a `0x`-prefixed 20-byte hex string."
5. PREREQUISITES: "Requires a paired Ledger (call `pair_ledger_live` first if `get_ledger_status` shows `paired: false`)."
6. RETURNS: "`{ handle, chainId: 1, to, valueWei, payloadFingerprint }` plus a `PREPARE RECEIPT` text block surfacing the verbatim args. The agent MUST pass the `handle` to `preview_send` next — without preview + the resulting `previewToken`, `send_transaction` refuses."
7. FAILURE MODES: "`WALLET_NOT_PAIRED` if no live session, `INVALID_INPUT` if `to`/`valueWei` malformed, `DEMO_MODE_REFUSED` if `VAULTPILOT_DEMO=true`."

The agent's routing decision between `prepare_native_send` (Phase 4) and `prepare_token_send` (Phase 6) is the highest-frequency error-class for downstream tool selection; the WHEN-NOT bullets explicitly disambiguate.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP boundary (agent → handler) | `to` + `valueWei` arrive as untrusted strings. JSON-schema validates type (string); handler validates shape (hex regex / bigint parse) before any state write. |
| Handler → handle-store | Handler is the only writer for the prepare-time record. Defense: store agent strings AND viem-typed values as distinct fields per `PrepareArgs` + `PreparedTx` (research § Pitfall 3) — the receipt block reads from agent strings; the fingerprint compute reads from viem-typed. Two-field split makes the verbatim guarantee structural, not a discipline. |
| Handler → session-manager | `getStatus()` is consulted for the SENDER address. Handler does NOT accept an agent-supplied `from` argument — the only sender is the paired Ledger's address. |
| Handler → response (text + structuredContent) | Response carries the `PREPARE RECEIPT` block (text) + the structured tuple (machine-readable). Both are surfaced; tampering with either is detectable by the other (defense in depth). |
