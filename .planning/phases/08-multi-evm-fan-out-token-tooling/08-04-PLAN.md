## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Agent → resolve_token (symbol input) | Symbol is case-insensitive string; table-driven lookup; NO RPC call. Unrecognized symbols refuse with INVALID_INPUT. NO way for the agent to inject a malicious address — the BRIDGED_VARIANTS table is the SOT; the agent can't add rows. |
| Curated BRIDGED_VARIANTS table → module load | Every address `getAddress`-checksummed at the literal site. Corrupted snapshot (single hex digit flipped) throws EIP-55 failure at module load. The table cannot silently ship a wrong address. |
| Agent → get_token_allowances (wallet input) | Wallet is regex-validated + `isAddress(wallet)` runtime check. The agent CAN query arbitrary wallets; this is read-only enumeration of public-state — no privilege required. NO leakage risk. |
| RPC `getLogs` response → enumeration | The RPC can return arbitrary event logs. Step 2 multicall cross-check filters revoked allowances. A compromised RPC could return phantom Approval events for tokens the user never interacted with; Step 2 would surface them only if `allowance(wallet, spender)` returns non-zero — which requires the malicious RPC to also fake the multicall response. Both layers would have to be compromised; in that case the `rpcDegraded: true` flag warns the user. |
| `[SET-LEVEL ENUMERATION]` block → v1.3 consumer | The block is the SOT artifact for v1.3 dispatch-allowlist enforcement. Drift in the template breaks the v1.3 parser at PR-review time. Plan 08-04 ships the canonical shape; future plans MUST extend additively (verbose footers, never breaking existing rows). |
