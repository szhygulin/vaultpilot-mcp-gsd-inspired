# Plan 01-03 — `--check` CLI subcommand

**Phase:** 1 (Server skeleton + install)
**Status:** Ready to execute
**Depends on:** 01-01 (needs the bin entrypoint routing stub)

## Goal

`npx -y vaultpilot-mcp --check` validates the install environment, prints `✓` / `⚠` / `✗` lines per check, and exits 0 on `ok` / `warn` or 1 on `error`. With `--json` it emits a structured `InstallEnvelope` for tooling consumption. After this plan ships, an agent can run `--check --json` at install time and adapt based on the envelope.

## Acceptance criteria

1. `--check` prints a human-readable doctor pass: header, one line per check, summary
2. Checks performed (Phase 1 set):
   - **node-version**: `process.versions.node >= 18.17` → `✓` / `✗`
   - **binary-spawn**: trivially true — if we got here, the binary spawned → `✓`
   - **wallet-connect-key**: `WALLETCONNECT_PROJECT_ID` env present → `✓` else `⚠ Phase 3+ requires this; not needed for read-only use`
   - **ethereum-rpc**: `ETHEREUM_RPC_URL` env present → `✓` else `⚠ Will fall back to PublicNode public RPC`
   - **config-file**: `~/.vaultpilot-mcp/config.json` parseable as JSON if present → `✓`/`⚠ malformed`/`(absent — auto-demo will run)`
3. `--json` flag suppresses human output and emits a single JSON object on stdout matching the `InstallEnvelope` shape:
   ```json
   {
     "envelope_version": 1,
     "status": "ok" | "warn" | "error",
     "checks": [
       { "id": "node-version", "level": "ok" | "warn" | "error", "message": "Node 20.10.0" }
     ],
     "metadata": { "vaultpilot_mcp_version": "0.0.0", "node_version": "20.10.0" }
   }
   ```
4. Exit code 0 on `ok` or `warn`; 1 on `error`
5. Vitest test exercises both human and `--json` output; covers all three status outcomes via env-var manipulation

## Implementation notes

- Status escalation: any `error` check → overall `error`. Any `warn` and no `error` → `warn`. All `ok` → `ok`.
- Don't read filesystem outside the user's home directory. Use `os.homedir()` not hard-coded paths.
- For the `wallet-connect-key` check: the env var is needed at Phase 3 for pairing, not Phase 1. The `⚠` message should say so explicitly so users don't think Phase 1 is broken.
- For the `ethereum-rpc` check: PublicNode is the documented free public fallback (per CONTEXT.md and REQUIREMENTS.md READ-05). Mention it by name in the message.
- For `config-file`: if absent, that's not a warning — it's the expected state for first-run users (auto-demo kicks in, per DEMO-07). The line should communicate "(absent — auto-demo will run)" without a warn level.
- JSON output goes to stdout. Human output goes to stdout too (the diagnostics logger is for SERVER stderr; the `--check` subcommand is interactive CLI output, not server diagnostics).

## Verify after execution

```bash
npm run build

# Human output, expected status: warn (no env vars set)
node dist/index.js --check
echo "exit: $?"   # should be 0 (warn is non-blocking)

# JSON output
node dist/index.js --check --json | jq .
# Should be valid JSON, status: "warn", envelope_version: 1

# Force ok status
WALLETCONNECT_PROJECT_ID=fake ETHEREUM_RPC_URL=https://eth.publicnode.com \
  node dist/index.js --check --json | jq .status
# Should print "ok"

# leave this to a follow-up test that mocks a failing check internally)
```
