# Plan 01-02 — MCP server framework

**Phase:** 1 (Server skeleton + install)
**Status:** Ready to execute
**Depends on:** 01-01 (needs the src/ skeleton + MCP SDK installed)

## Goal

A boot-able MCP server that responds to the `initialize` MCP protocol message with a tool list (initially empty) and a server-level `instructions` field carrying the one-paragraph self-description. After this plan ships, you can register the server with Claude Code and `claude mcp list` shows it as connected.

## Acceptance criteria

1. `src/server.ts` exports `startServer()` that constructs an MCP `Server` and wires `StdioServerTransport`
2. The `Server` declares its `instructions` field per CONTEXT.md: one paragraph naming what tools do, the security-model anchor sentence, and a link to SECURITY.md (which doesn't exist yet — link `./SECURITY.md` regardless; v1.3 will populate it)
3. A tool registration framework exists: `registerTool(name, description, inputSchema, handler)` callable from elsewhere in the codebase. Handler signature returns the tool's structured result. Handler errors get wrapped as MCP error responses (not thrown to the transport).
4. Stderr is the only diagnostic output channel; stdout is the MCP transport. A logger helper (`src/diagnostics/logger.ts`) enforces this.
5. Tool registration framework supports `description >= 100 chars` validation (warn, don't refuse) — descriptions are the agent's routing prompt, short ones cost the agent.
6. `src/index.ts` invokes `startServer()` when no recognized subcommand is present
7. Vitest test: spawn the server in-process, send `initialize`, assert response carries `instructions` field with the SECURITY.md link

## Implementation notes

- MCP SDK 1.x uses `Server` from `@modelcontextprotocol/sdk/server/index.js`. The `instructions` field is set in the server's metadata at construction time.
- `StdioServerTransport` from `@modelcontextprotocol/sdk/server/stdio.js`.
- Tool registration: use `server.setRequestHandler(ListToolsRequestSchema, ...)` and `CallToolRequestSchema` with a registry lookup. Don't try to be clever — just a Map of name → { description, inputSchema, handler }.
- Stderr logger is dead-simple: `process.stderr.write(\`[\${level}] \${msg}\\n\`)`. No log libraries.
- For the initialize-handshake test: use the SDK's `Client` + an in-memory transport pair (the SDK ships an `InMemoryTransport`).
- `instructions` content: keep tight. Three sentences max. State what tools do (read on-chain positions + prepare Ledger-signed transactions), name the security anchor (the user trusts what the Ledger screen shows), point at SECURITY.md.
