# Plan 02-04 — ENS resolver + `get_token_balance` + `get_transaction_status`

**Phase:** 2 (Ethereum read-only portfolio)
**Status:** Ready to execute
**Depends on:** 02-01 (uses `getEthereumClient()`)

## Goal

Four small standalone read tools: ENS forward + reverse, single-token balance, tx-status polling. None of these aggregate; they're the building blocks the agent calls when the user asks "what's my USDC balance" or "did my tx land?".

## Acceptance criteria

1. `src/ens/resolver.ts` wraps `viem.getEnsAddress` + `viem.getEnsName` (Universal Resolver under the hood)
2. `src/tools/resolve_ens_name.ts` registers `resolve_ens_name({ name })` → `{ address: "0x..." | null }`; null when name doesn't resolve
3. `src/tools/reverse_resolve_ens.ts` registers `reverse_resolve_ens({ address })` → `{ name: "..." | null }`; null when no primary name set
4. `src/tools/get_token_balance.ts` registers `get_token_balance({ wallet, tokenAddress })` → `{ balance, decimals, symbol, balanceUsd?, priceUnknown? }`
5. `src/tools/get_transaction_status.ts` registers `get_transaction_status({ txHash })` → `{ status: "pending" | "success" | "reverted", blockNumber?, gasUsed? }`; uses `viem.getTransactionReceipt` + falls back to `getTransaction` for pending detection
6. `src/tools/register-all.ts` adds 4 import lines for the new tool modules
7. Tests for each tool cover the happy path + the null/pending edge case
8. `npm run typecheck` + `npm run build` + `npm test` all pass

## Out of scope (defer)

- ENS profile records (avatar, etc.) — out of v1.x
- USD pricing on `get_token_balance` is OPTIONAL — if 02-03's `getPrices` exists by the time this lands, wire it; if not, omit `balanceUsd` and `priceUnknown` from the response (the field becomes `undefined`, no error)
- Multi-chain ENS — ENS is canonical on Ethereum mainnet; L2 ENS is Phase 8
- Tx history (`get_transaction_history`) — Phase 6+
