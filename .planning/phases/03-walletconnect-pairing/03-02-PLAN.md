## VERBATIM VERIFY-ON-DEVICE block (PAIR-03 source-of-truth string)

This text is FROZEN. It must appear in `src/tools/pair_ledger_live.ts` exactly once, as a `const VERIFY_ON_DEVICE_TEMPLATE: string` exported from the module (so tests can `import { VERIFY_ON_DEVICE_TEMPLATE }` and `toContain`-assert against it — per the format-fanout-regex-sync rule, the test must NOT inline the string).

```text
VERIFY-ON-DEVICE
  Address: {ADDRESS}
  Session topic (last 8): {SESSION_TOPIC_LAST8}

In Ledger Live → Settings → Connected Apps:
  - Confirm the address shown for this app matches the address above.
  - Confirm the session topic (last 8 hex chars) matches.
  - If either doesn't match, DO NOT proceed with any signing flow.
    Treat it as a tamper signal and re-pair with force: true.
```

`{ADDRESS}` and `{SESSION_TOPIC_LAST8}` are template placeholders substituted at runtime with the paired values. The substitution is `template.replace("{ADDRESS}", address).replace("{SESSION_TOPIC_LAST8}", sessionTopicLast8)` — NOT a template literal `\`...\`` (the template constant must be a plain string so it ships unchanged into the test file's `toContain` check).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP tool response → agent | Structured `errorCode` is the agent's only routable signal; free-text errors leak intent and force the agent to parse human prose |
| Tool input schema → handler | JSON-schema validation at MCP boundary; `force` is `boolean` only, no defaults exploited via type coercion |
| Tool response text → user | VERIFY-ON-DEVICE block is the user's cross-check artifact; tampering removes the block — partially mitigated by the structured-content shape contract |
