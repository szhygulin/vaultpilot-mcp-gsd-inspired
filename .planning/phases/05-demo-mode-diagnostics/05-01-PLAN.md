## Auto-demo first-call side effect

When `resolveDemoMode()` resolves to `mode: "auto-demo"`, the resolver MUST seed the active persona at the same first-call site:

```typescript
if (!cached) {
  // ... resolution ...
  cached = { mode: "auto-demo" };
  // Q-AUTO-DEMO-PERSONA-DEFAULT: seed whale so the first read tool works
  // out of the box without requiring set_demo_wallet first. The seed is
  // idempotent — if a future test pre-seeds a different persona via
  // setActivePersona() before the resolver fires, the resolver's check
  // `if (getActivePersona() === null)` lets the test value win.
  if (getActivePersona() === null) {
    setActivePersona("whale");
  }
}
```

Why guard with `getActivePersona() === null`: test isolation. A test that pre-seeds `setActivePersona("stable-saver")` then triggers auto-demo should keep stable-saver active; the seed fires only when no persona is set. Implicit contract: production code never calls `setActivePersona` before `isDemoMode()` resolves the first time, so the guard is a defensive no-op in prod.

## Strict-literal env-var arm (Q-STRICT lock)

```typescript
const envRaw = process.env.VAULTPILOT_DEMO;
if (envRaw !== undefined) {
  if (envRaw === "true") { cached = { mode: "env-on" }; return cached; }
  if (envRaw === "false") { cached = { mode: "env-off" }; return cached; }
  log("error", `VAULTPILOT_DEMO must be literal "true" or "false"; got "${envRaw}". Refusing to boot.`);
  process.exit(1);
}
```

`process.exit(1)` returns `never` (TS lib.dom.d.ts). The resolver function's return type stays `ResolutionResult` because the path past `process.exit` is unreachable. No type-level fallthrough issue.

**The strict-literal regression anchor at `test/pair-ledger-live.test.ts:207-226`** (`'True'` capitalized does NOT trigger refusal) is preserved through `_resetDemoModeForTesting()` discipline AND the literal-`"true"` arm staying byte-identical. That test sets `process.env.VAULTPILOT_DEMO = 'True'` then calls `pair_ledger_live`. Under the new chain, `'True'` is NOT one of the two accepted literals → the resolver would exit. **Critical:** that test must EITHER call `_resetDemoModeForTesting()` in `beforeEach` (it already does — line ~46) AND set the env var to a recognized literal OR delete it. Under the new chain that test now FAILS because the resolver process.exits.

**Migration plan for the regression anchor:** the Phase 3 test asserted "capital T is not a demo signal" — the new contract is stricter: capital T is a BOOT FAILURE. The semantic is preserved (capital T is not demo); the failure mode tightens (refuses to boot rather than silently treating as real-mode). Update `test/pair-ledger-live.test.ts:207-226` in this plan:
- Wrap the body in a `expect(() => isDemoMode()).toThrow(/VAULTPILOT_DEMO must be literal/)` or similar — but `process.exit(1)` isn't throwable. Instead: **isolate the test by setting the env var to a recognized literal BEFORE the offending case**, OR **convert the regression assertion to test the NEW failure mode**: that an unrecognized literal triggers a process.exit-1 call (mock `process.exit` for the duration of the test, assert it was called with 1, assert stderr message matches).

The plan ships BOTH changes in the same commit — the resolver behavior change AND the test update. The semantic protection survives: `"True"` is not a valid demo signal. The mechanism changes: silent fallthrough → loud exit. **This is a Phase-3-test-suite migration noted explicitly in the PR description.**

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Process env → resolver | `VAULTPILOT_DEMO` is untrusted host-supplied input. Strict-literal check (Q-STRICT) defends against silent misconfig where `"1"` / `"yes"` / `"True"` would otherwise leak the wrong mode. Any unrecognized value refuses to boot — loud failure beats silent wrong-mode. |
| Filesystem → resolver | `~/.vaultpilot-mcp/config.json` is host-trusted (the user wrote it). A compromised host that flipped `demo: true` → `demo: false` (or vice versa) is OUT OF SCOPE — host-FS compromise is documented in PROJECT.md as the host-trust boundary; defense is the Ledger device, not the MCP. Malformed JSON → boot refusal (not silent fallthrough) so the operator notices. |
| Resolver → consumers (Phase 3+4 tools) | `isDemoMode()` is the single predicate every Phase 3+4 refusal path reads. The resolver caches at first call so a runtime `delete process.env.VAULTPILOT_DEMO` between tool calls does NOT silently flip mode — `_resetDemoModeForTesting()` is the only way to recompute. Defends against accidental cross-test contamination. |
| Set-demo-wallet → mutable state | `setActivePersona` mutates module-scoped `let`. Refusal with `WRONG_MODE` when `!isDemoMode()` prevents state mutation across the demo/real boundary (Q-WRONG-MODE). |
| Schema enum → handler | `inputSchema.properties.persona.enum: ["whale", "defi-degen", "stable-saver", "staking-maxi"]` is the protocol-boundary gate; Phase 4's `AjvJsonSchemaValidator` at `src/server.ts:55-66` runs BEFORE the handler. Unknown slug → `-32602 InvalidParams` at the JSON-RPC layer; handler never reached. In-handler `PERSONAS.find` re-check is defense-in-depth. |
