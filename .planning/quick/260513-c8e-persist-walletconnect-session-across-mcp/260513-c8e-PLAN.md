<objective>
Persist the WalletConnect v2 session across MCP restarts so the user does not have to re-pair Ledger Live every time their MCP client (Claude Code / Cursor / Desktop) reloads the server. Default to persistent storage at `~/.vaultpilot-mcp/wc-storage/` with `0o700` perms; opt out via `VAULTPILOT_WC_STORAGE=memory`. Tests stay hermetic (opt-out by default). Resolves GitHub issue #25.

Purpose: Phase 3 pinned `:memory:` as a security-conservative default ("fresh process = fresh pair"). In real use, MCP clients restart often enough that pairing every cold-boot is the friction that caps the product. The trust anchor is the Ledger screen — the WC session symmetric keys it persists are session-scoped pairing artifacts, not wallet private keys (those never leave the device). The trade-off (filesystem-trust assumption + lower per-cold-boot user-confirm friction) is acceptable on a primary developer machine and is documented + opt-out-able.

Output: One MCP-side change (wc-storage helper + walletconnect-client wiring + diagnostics surface + force-re-pair teardown + test hermeticity + SECURITY.md). The WC session DB plumbing changes behavior; the cryptographic-binding chain (payloadFingerprint → previewToken → userDecision → Ledger screen) is UNCHANGED — persistence only affects the WC pairing handshake.

User-visible behavior change: mode is selected at process start; switching modes requires an MCP restart (the singleton client is captured at first SignClient.init). The plan does NOT migrate an in-flight :memory: session to disk at runtime.
</objective>

From `node_modules/@walletconnect/keyvaluestorage/dist/index.cjs.js` (runtime, NOT just types):

The SDK consumes `storageOptions: { database?: string; table?: string }`. Internally:
- The literal sentinel `":memory:"` selects an in-memory `unstorage` backing (no fs writes).
- ANY OTHER string is resolved as `dbName = database || table || "walletconnect.db"`, then passed to `unstorage/drivers/fs-lite` as `{ base: dbName }`.
- **`fs-lite` treats `base` as a DIRECTORY**, not a single file. Each key becomes a file inside that directory. So `wc-storage.db` is a misnomer if read as a single-file name — `wc-storage/` (a directory) is the accurate mental model. The plan uses `~/.vaultpilot-mcp/wc-storage/` as the path (no trailing `.db` extension is necessary; the directory contains the WC keys-as-files).
- On first write, the SDK calls `existsSync(dbName)` and (if not a directory) renames it to `${dbName}.to_migrate` before creating the dir. Harmless on a clean install.
